# Правила написания документации на конвенции (Conventions Documentation Guidelines)

## Общие правила

- Документ должен быть кратким, точным и легко читаемым.
- Фразы и формулировки как "пули", попадают быстро и точно в цель: короткие, четкие, понятные, без воды и дублирования.
- Используем структурированный формат с заголовками и одинаковыми разделами.  
- Каждый документ описывает **один подход** или **одну сущность**.  
- В тексте используем ссылки на официальные источники (Wikipedia, Symfony docs, книги по DDD и Clean Code).  
- Язык: **русский** (с английским термином в скобках при первом упоминании).  
- Важно: **без воды**, только то, что поможет программисту применить подход правильно.  
- Код пишем на PHP 8.4 с использованием Symfony 7.3, строго типизированный.  

## Обязательная структура документа

Каждый документ должен содержать следующие разделы:

1. **Заголовок**  
   - На русском + в скобках английский термин.  
   - Пример:  
     ```
     # Спецификация (Specification pattern)
     ```

2. **Определение**  
   - Краткое описание подхода.  
   - Указываем ссылку на источник, если это паттерн или термин (например, Gang of Four, DDD).

3. **Общие правила**  
   - Четкие, нумерованные или маркированные пункты.  
   - Указываем: можно/нельзя, как именовать классы, как внедрять в код.  

4. **Зависимости** (если применимо)  
   - Что можно внедрять в класс.  
   - Что запрещено (например, зависимости из других слоёв).  

5. **Расположение** (если применимо)
   - Указываем слой (Domain, Application, Infrastructure, Presentation).  
   - Путь в иерархии папок.  
   - Пример пути.  

6. **Как используем**  
   - Правила внедрения.  
   - Ограничения (где допустимо, где запрещено).  

7. **Пример**  
   - Минимальный рабочий пример на PHP (короткий и понятный).  
   - В примере демонстрируются **правильные практики**.  

## Форматирование

- Используем **Markdown**.  
- Кодовые блоки:  
  ```php
  <?php
  declare(strict_types=1);
  ```
- ❗ Запрещено использовать псевдокод — всегда реальный PHP-пример, взятый из текущего проекта и упрощенный.  
- Комментарии в примере минимальны, код должен быть самодокументируемым.
