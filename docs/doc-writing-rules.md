# Правила написания документации (Documentation Writing Guidelines)

## Принципы

- Документ должен быть кратким, точным и легко читаемым.
- Один документ — одна тема/процесс/сущность; не смешивай разные правила в одном файле.
- Язык: русский (английский термин — в скобках при первом упоминании).
- Технические сущности называй на английском: `branch`, `commit`, `Pull Request`, классы, методы, файлы, команды.
- Пиши проверяемо: избегай двусмысленности, “примерно”, “как-нибудь”, “по возможности”.
- Не описывай то, чего нет в проекте (процессы, инструменты, ссылки); не добавляй неработающие ссылки.
- Ссылайся на официальные источники (Wikipedia, Symfony docs, книги по DDD/Clean Code) и на документы внутри `docs/`.
- При добавлении нового документа добавь ссылку на него в соответствующий индекс/оглавление (например, `docs/README.md` или `index.md` в разделе).
- Примеры кода — реальный PHP 8.4 / Symfony 7.3, строго типизированный; псевдокод запрещён.

## Структура документа (минимум)

1. **Заголовок**
   - На русском + (английский термин).

2. **Определение / цель**
   - 1–3 предложения: что это и зачем нужно.

3. **Границы ответственности**
   - Что документ регулирует, а что — нет; куда ссылаться вместо дублирования (например, на `docs/conventions/*`).

4. **Правила / шаги**
   - Чёткие, короткие пункты “можно/нельзя/нужно сделать”.
   - Если это процесс — пошаговая инструкция.

5. **Пример** (если применимо)
   - Минимальный пример: команда, шаблон текста, небольшой фрагмент кода.

6. **Чеклист** (опционально)
   - Короткий список проверок для самопроверки/ревью.

## Форматирование

- Используй Markdown.
- Для команд/путей/идентификаторов используй inline code: `make check`, `docs/git-workflow/code-review.md`.
- Для команд и примеров кода используй блоки кода.
- В коде:
  ```php
  <?php
  declare(strict_types=1);
  ```

````markdown

## Границы ответственности

- ...

## Правила / шаги

- ...

## Пример

```bash
# ...
```

## Чеклист

- [ ] ...
````
